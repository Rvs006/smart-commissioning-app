# Outcome

Ship one verified release that makes a UDMI validation run read the same in the
application and in every generated report. The work covers the field review
document, both supplied PDF reports, the embedded screenshots, the executable,
the repository documentation, and the release pipeline.

The supplied PDFs are evidence from two different immutable runs. The later
report is a 22-asset subset of the earlier 108-asset run, so their totals must
not be compared as if one were a rerender of the other. The release acceptance
test will compare the GUI and report generated from the same run ID and the same
active row selection.

# Evidence summary

- The Word review contains 33 screenshots and 27 numbered or lettered actions.
- The 460-page report contains 108 expected assets, 103 observed assets, 324
  expected payloads, and 1,430 findings.
- The 72-page report contains 22 expected assets, 21 observed assets, 66
  expected payloads, and 211 findings.
- The 22 shared assets have the same 210 asset-level findings in both reports.
  The shorter PDF is a filtered run, not evidence that those findings changed.
- Current PDF detail rows truncate values and then repeat the same finding as
  loose prose. That duplication is the main reason the larger report reaches
  460 pages.
- The GUI already computes the exact visible payload rows. The report request
  currently sends only source run IDs, so it silently rebuilds the full run.

# Product decisions

1. Expected and observed asset metrics remain register-scoped. An unexpected
   publisher is counted separately and never inflates Observed Assets.
2. Unexpected devices are visible and filterable in the GUI, but excluded from
   the main validation-result tables and compliance calculation.
3. A filtered report is defined by exact selected payload rows. Filter labels
   are stored only as readable provenance and are never trusted as counts.
4. Run-level capture failures may appear in fault detail, but they cannot create
   synthetic assets in the Fault Matrix.
5. Project and Site come from the imported register. No real customer, site,
   device, network, or personnel identifiers are committed to this public
   repository.
6. Seasonal RFC 3339 offsets such as `+00:00` and `+01:00` remain valid. The new
   consistency check compares lexical style, not timezone value.
7. The DBO unit vocabulary is pinned to an upstream commit and shared by import
   and validation code. `parts_per_billion` and `parts_per_million` remain
   distinct units.

# Work packages

## P1: Validation correctness

Files include `core/smart_commissioning_core/udmi_schema.py`,
`core/smart_commissioning_core/udmi_validation.py`, shared unit resources, and
core tests.

- Report missing `timestamp` and other required fields even when `version` is
  also absent and the register provides the authoritative schema version.
- Add a pinned, shared Google Digital Buildings unit vocabulary.
- Accept `parts_per_billion` and the `ppb` alias without treating it as ppm.
- Attach `match_basis=units` to unit findings so the UI can flag the exact
  `units` field.
- Add a register-policy issue for a missing `system.location` container when
  the imported row expects a site or room.
- Generate expected metadata with `system.location.room` while continuing to
  accept the existing `section` shape during transition.
- Detect mixed timestamp notation inside one payload with the required opening
  sentence: `Inconsistent timestamp formats in the payload.`
- Add regression tests for `Z`, `+00:00`, and `+01:00` in state, metadata, and
  pointset payloads.
- Renumber final UDMI findings once, after validation, so issue IDs are unique
  within an immutable run.

Acceptance:

- Empty payloads do not hide required-field findings.
- Valid ppb data produces no vocabulary warning.
- Unit-only findings highlight the unit line rather than the point name.
- Offset changes do not create false timestamp errors.
- Mixed fractional precision or mixed `Z` and signed-offset notation creates
  one precise payload-formatting issue.

## P2: Unexpected-device measurement

Files include live UDMI capture, validation-result aggregation, API contracts,
and GUI result rows.

- Derive a bounded register topic root and observe publishers inside that scope.
- Never subscribe to a bare broker-wide `#` solely for this metric.
- Deduplicate unexpected publishers by normalized topic root.
- Persist numeric count, display identities, topics, and a measurement-complete
  flag in validation summary schema 1.1.
- Keep unexpected publishers out of registered asset success and validation
  issue totals.
- Add a Category filter with Expected and Unexpected options.

Acceptance:

- Supporting Metrics contains a numeric Unexpected Devices value.
- An unexpected row is visible and filterable in the GUI.
- No unexpected row appears in the report validation schedule, Fault Matrix, or
  validation detail.
- System filtering returns zero unexpected devices because an unregistered
  publisher has no trusted register system assignment.

## P3: Metric parity and exact report scope

Files include `ModulePage.tsx`, API client and schemas, run service, validation
summary helpers, report model, and parity tests.

- Make all headline metrics respond to text, verdict, topic, system,
  observation, category, and combined filters.
- At report submission, capture exact visible tuples of source run ID, asset ID,
  and payload type.
- Persist a bounded UDMI scope with the report run.
- Validate every selected tuple against the terminal source summary. Reject
  cross-run, unknown, duplicate, and unsupported legacy selections.
- Snapshot the server-derived report model when the report run is created so
  later downloads never re-read mutable source records.
- Recompute asset, payload, issue, and system metrics from the selected rows.
- Count Payloads With Issues only when the expected payload was received.
- Add a separate Not Received metric.
- Scope structured `findings.json` output to the same selection.

Acceptance:

- A no-match filter produces a valid zero-row report rather than a full report.
- A filtered GUI and its report have identical totals and retained issue IDs.
- Clearing filters restores the immutable full-run summary.
- The server never accepts client-supplied totals.

## Adversarial closure added during implementation

The final integration pass added regressions for nine boundary cases that were
not visible in the supplied reports alone:

- Non-expected payload rows stay visible as evidence but never enter an exact
  expected-payload report selection or mark a registered asset observed.
- A present 1.0 or 1.1 summary with malformed nested rows fails closed instead
  of selecting the pre-contract renderer.
- Non-expected rows cannot prevent a complete expected selection from retaining
  asset-level or run-level findings.
- The GUI applies the same received-only issue-count normalization as reports
  when it reads a stored 1.0 summary.
- Retired unexpected-publisher findings are removed by stored issue ID, including
  older rows whose category was recorded as `other_issues`.
- A complete filtered source retains run-wide findings, while a partial source
  does not claim them.
- Filtered asset observation and evidence time are recomputed from the selected
  expected payloads.
- Evidence timestamps are ordered by RFC 3339 instant, so valid mixed offsets do
  not change which observation is latest.
- A separate retained-topic budget prevents observation-only wildcard traffic
  from consuming the expected-validation capture cap.
- Release publication accepts only the successful versioned workflow run from
  `main`, targets its exact commit, and verifies tag, asset, notes, and version
  metadata before reporting success.

# Release stop conditions

The release stops if any test or CI job fails, if GUI and report metrics differ
for the same selected rows, if report cells truncate live content, if public
fixtures contain real field identifiers, or if the final branch/tag alignment
cannot be proven. A fix is complete only after those conditions are clear.
