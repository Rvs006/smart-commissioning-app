# Phase 5 Live-Surface Inventory — authoritative list of every live-untested path

Every row below is a code path that touches real hardware, a real broker, a real network, or real infra and therefore **could not be exercised in dev/CI** (no BACnet device, no live broker, no Redis/Postgres, no remote hub, no Docker daemon). Each engine/transport file carries its own honesty banner saying so (e.g. `ip_scan.py` module docstring "ON-SITE VALIDATION REQUIRED", `bacnet_discovery.py` `Bacpypes3Backend` "!!! NEVER INTEGRATION-TESTED !!!", `mqtt_transport._wrap_tls` "the real TLS handshake against a live broker is on-site-untested", `sync.py` "A real remote hub ... is NOT implemented here ... live_untested").

Conventions used in "How to trigger on-site":
- API routes are mounted under `/api/v1` behind `require_auth` (set `AUTH_MODE=api_key`, send `X-API-Key`). Create-run routes need RBAC **engineer+** (`require_role(Role.ENGINEER)`); read routes need **viewer+**.
- Broker/scan params live in the run's `parameters` JSON. Broker host resolves from `parameters.broker_host` first, else stored config (`build_mqtt_connection_settings`, `mqtt_settings.py`). TLS turns on when `parameters.use_tls` is set or `broker_port == 8883`.
- **Authorization contract** (verbatim from `engines/safety.py`): a real scan/publish needs EITHER `parameters.authorized == true` OR `parameters.scan_authorization = {"authorized": true, "authorized_by": "<who>"}`. A `dry_run == true` request is side-effect-free and needs no authorization.

## Cross-cutting safety sequence (apply to every "live" row above)
1. **Dry-run** (where supported: #1, #3, #4) → confirms plan, zero I/O.
2. **Lab / non-production segment** with `scan_authorization` set.
3. **Production building** only in a change window with site authorization, after the throttle (#1), lab-validated BACnet adapter (#3), and lab-validated rollback (#9) each pass.

## Honesty notes
- Rows #1–#9, #14 perform **no** real I/O in this dev env; their pure logic (target expansion, aggregation, record building, throttle, cancel, auth gate, dry-run, credential sanitization) is unit-tested against fakes/loopback only — see each module's docstring and `core/tests/test_{ip_scan,mqtt_discovery,sync}.py`.
- The existing `docs/phase5-onsite-validation.md` checklist references `# UNVERIFIED:` markers in `bacnet_discovery.py`; the current source actually uses `# VERIFIED against bacpypes3 …` plus explicit "BEST-EFFORT"/"KNOWN LIMITATION" notes (vendorID attr name, object-list APDU chunking). The substance is the same — those two items are the real unverified-against-hardware points to confirm in row #3.
